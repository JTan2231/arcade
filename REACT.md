# React Frontend Migration

This document describes replacing Arcade's current plain static frontend with a
React frontend built with Bun and Vite.

The backend remains Go. Go continues to own process startup, migrations,
session auth, JSON APIs under `/api/*`, and static asset serving from the
embedded `web/static` filesystem.

## Current State

Arcade currently ships as one Go binary with an embedded static frontend:

- `web/static.go` embeds `web/static` with Go `embed.FS`.
- `internal/app/server.go` serves the embedded static files at `GET /`.
- `web/static/index.html`, `web/static/app.js`, and `web/static/styles.css`
  implement the browser app directly.
- The browser calls `/api/*` with `fetch` and same-origin cookies.
- There is no frontend package manager, bundler, transpiler, or frontend build
  step.

The currently implemented static UI covers session bootstrap, login, signup,
logout, group listing, group creation, and selected-group state. The API surface
is broader than the UI and should not need to change for this migration.

## Target State

Use React as the UI implementation, Vite as the frontend dev/build tool, and Bun
as the package manager and script runner.

```text
Browser
  -> Go-served static React build from /
  -> fetch('/api/...') JSON requests
  -> Go auth middleware loads arcade_session cookie
  -> Go handler
  -> Postgres
```

Bun is not the production web server. Vite is not the production web server.
Production remains the Go binary serving embedded static files.

## Tool Responsibilities

`Go`

- Runtime server.
- API server.
- Static file server.
- Production deployable artifact.

`Bun`

- Installs frontend dependencies.
- Writes and enforces `bun.lock`.
- Runs frontend scripts in local development and CI.

`Vite`

- Runs the local React dev server.
- Bundles React, TypeScript, and CSS into static production assets.
- Emits build output into `web/static`.

`React`

- Owns browser state, rendering, form handlers, and UI composition.
- Does not own authentication storage. The session remains server-side and
  cookie-backed.

## Proposed Layout

```text
web
  static.go
  static/
    index.html              # generated by Vite build
    assets/                 # generated hashed JS/CSS assets
  frontend/
    package.json
    bun.lock
    index.html              # Vite source entry
    vite.config.ts
    tsconfig.json
    tsconfig.node.json
    src/
      main.tsx
      App.tsx
      api.ts
      types.ts
      styles.css
      components/
        AuthView.tsx
        GroupsPanel.tsx
        Toast.tsx
```

`web/static` remains the Go embed target. `web/frontend` becomes the editable
React source tree.

## Generated Asset Policy

Two policies are possible.

Recommended policy:

- Treat `web/static` as generated output.
- Commit React source and `web/frontend/bun.lock`.
- Do not hand-edit files in `web/static`.
- Require `bun run build` before `go build`, `go run`, and CI Go build steps.

Tradeoff: a fresh checkout must run the frontend build before the Go binary can
embed the newest frontend assets.

Alternative policy:

- Commit Vite output in `web/static`.
- Keep `go build ./cmd/arcade` working without Bun after checkout.
- Add CI enforcement that rebuilding the frontend produces no diff.

Tradeoff: generated hashed assets create noisier commits and stale output is a
constant risk.

For this migration, prefer the recommended policy unless Go-only builds from a
fresh checkout are a hard requirement.

## Vite Configuration

Vite should build from `web/frontend` into `web/static`.

```ts
// web/frontend/vite.config.ts
import { defineConfig } from "vite";
import react from "@vitejs/plugin-react";

export default defineConfig({
  plugins: [react()],
  base: "/",
  build: {
    outDir: "../static",
    emptyOutDir: true,
  },
  server: {
    proxy: {
      "/api": "http://localhost:8080",
    },
  },
});
```

The dev proxy keeps browser API calls same-origin from the React dev server's
perspective. The production build still calls `/api/*` on the Go server.

## Package Scripts

Use Bun to run scripts. A minimal `web/frontend/package.json` should expose:

```json
{
  "scripts": {
    "dev": "vite",
    "build": "tsc --noEmit && vite build",
    "preview": "vite preview",
    "check": "tsc --noEmit"
  },
  "dependencies": {
    "react": "...",
    "react-dom": "..."
  },
  "devDependencies": {
    "@types/react": "...",
    "@types/react-dom": "...",
    "@vitejs/plugin-react": "...",
    "typescript": "...",
    "vite": "..."
  }
}
```

Exact dependency versions should be recorded in `bun.lock`. CI should use
`bun ci`, not `bun install`, so dependency resolution is reproducible.

## Setup Commands

Initial scaffold:

```sh
mkdir -p web/frontend
cd web/frontend
bun create vite . --template react-ts
bun install
```

After scaffolding, adjust `vite.config.ts` so `build.outDir` is `../static` and
the dev server proxies `/api` to the Go server.

Local backend:

```sh
go run ./cmd/arcade
```

Local frontend dev server:

```sh
cd web/frontend
bun run dev
```

Production-style embedded build:

```sh
cd web/frontend
bun ci
bun run build
cd ../..
go run ./cmd/arcade
```

## Authentication Contract

Do not introduce frontend token storage.

The backend issues an HttpOnly `arcade_session` cookie. React cannot and should
not read that cookie. React should discover session state by calling:

```http
GET /api/auth/session
```

All API calls should include same-origin credentials:

```ts
const response = await fetch(path, {
  credentials: "same-origin",
  headers: {
    "Content-Type": "application/json",
  },
  ...options,
});
```

On `401`, the React app should clear client state and show the auth view. It
should not clear cookies directly; logout remains:

```http
POST /api/auth/logout
```

API errors are JSON objects shaped as:

```json
{ "error": "message" }
```

The React API helper should preserve that error behavior.

## React State Model

The initial React port should preserve the current static app's behavior before
adding new product surface area.

Current static concern:

```text
state.authenticated
state.groups
state.selectedGroupId
auth mode: login/signup
toast message
```

React equivalent:

```text
App
  sessionUser: User | null
  authStatus: "checking" | "anonymous" | "authenticated"
  groups: Group[]
  selectedGroupId: string | null
  toast: string | null
```

On app mount:

1. Call `/api/auth/session`.
2. If authenticated, call `/api/groups`.
3. Select the first active group if no selected group exists.
4. If unauthenticated, render the auth view.

## API Types

Define TypeScript types matching `internal/app/types.go`. Start with the types
needed by the current UI:

```ts
export type User = {
  id: string;
  email: string;
  username: string;
  display_name: string;
  avatar_url?: string;
  created_at: string;
  updated_at: string;
};

export type Group = {
  id: string;
  name: string;
  slug: string;
  description?: string;
  visibility: "public" | "invite_only" | "private";
  created_by_user_id: string;
  my_role?: "owner" | "admin" | "member";
  my_status?: "invited" | "active" | "removed" | "left";
  created_at: string;
  updated_at: string;
};
```

Add broader API types only as the React UI starts using those endpoints.

## Component Boundaries

Start with a direct port:

```text
App
  AuthView
    LoginForm
    SignupForm
  Shell
    Header
    GroupsPanel
  Toast
```

Suggested responsibilities:

- `api.ts`: fetch wrapper, typed API functions, error normalization.
- `types.ts`: API response/request types.
- `AuthView.tsx`: auth tabs, login form, signup form.
- `GroupsPanel.tsx`: group creation form and group list.
- `Toast.tsx`: transient status messages.
- `App.tsx`: bootstrap, top-level state, and view switching.

Avoid adding a global state library in the first port. React state is enough for
the current surface.

## Routing

Do not add browser-history client routes in the first migration.

The current Go static handler serves files directly from the embedded filesystem.
Without a server-side SPA fallback, direct browser requests to paths like
`/groups/123` would look for a static file at that path and fail.

Acceptable first-phase options:

- Keep the app route-less and state-driven.
- Use hash routes such as `/#/groups/123` if deep links become necessary before
  backend static fallback work.

If history routes are required later, add a Go static fallback that serves
`index.html` for non-API paths that do not match a real static asset. That is a
separate backend-serving change and is intentionally out of scope for the first
React replacement.

## CSS Strategy

Use plain CSS first.

The current app has a small stylesheet and no design system dependency. A direct
React port can move styles to `web/frontend/src/styles.css` and import it from
`main.tsx`.

Avoid adding Tailwind, CSS-in-JS, or a component library during the initial
migration unless there is a concrete UI requirement that justifies it.

## CI Changes

`ci.sh` should become responsible for building the frontend before compiling the
Go binary.

Current frontend CI check:

```sh
node --check web/static/app.js
```

React migration replacement:

```sh
if ! command -v bun >/dev/null 2>&1; then
  printf 'bun is required but was not found in PATH\n' >&2
  exit 1
fi

section "Installing frontend dependencies"
(cd web/frontend && bun ci)

section "Checking frontend"
(cd web/frontend && bun run check)

section "Building frontend"
(cd web/frontend && bun run build)
```

Then run existing Go checks:

```sh
go mod tidy -diff
go vet ./...
go test ./...
go build -trimpath -o "$tmp_dir/arcade" ./cmd/arcade
```

The frontend build must run before `go build` so Go embeds the current
`web/static` output.

## Git Ignore Changes

At minimum:

```gitignore
web/frontend/node_modules/
```

If `web/static` is treated as generated output and not committed, ignore the
generated files carefully. Keep the directory itself present if needed by local
tooling.

```gitignore
web/static/*
!web/static/.gitkeep
```

If generated output is committed, do not ignore `web/static`.

## Migration Sequence

1. Create `web/frontend` with Vite React TypeScript using Bun.
2. Configure Vite to build to `web/static` and proxy `/api` to the Go server in
   development.
3. Add `api.ts` with same-origin fetch behavior and current JSON error handling.
4. Add TypeScript API types for `User` and `Group`.
5. Port the current static app behavior into React:
   - session bootstrap,
   - login,
   - signup,
   - logout,
   - group list,
   - group creation,
   - selected group state,
   - toast messages.
6. Move CSS into the React source tree and preserve the existing visual behavior
   unless there is a separate redesign task.
7. Replace `ci.sh` frontend syntax check with Bun install/check/build steps.
8. Decide and enforce the generated asset policy for `web/static`.
9. Update `README.md`, `docs/architecture.md`, and `docs/tech-stack.md`.
10. Run `./ci.sh`.

## Acceptance Criteria

The migration is complete when:

- `cd web/frontend && bun ci` succeeds.
- `cd web/frontend && bun run build` emits `web/static/index.html` and hashed
  assets.
- `./ci.sh` succeeds.
- `go run ./cmd/arcade` serves the built React app from `/`.
- Login works.
- Signup works.
- Session bootstrap keeps an already signed-in user in the app.
- Logout works.
- Groups load from `/api/groups`.
- Creating a group posts to `/api/groups`, refreshes the list, and selects the
  new group.
- No bearer tokens, localStorage session tokens, or frontend-readable session
  cookies are introduced.

## Main Risks

- Bun becomes a required build dependency.
- If `web/static` is not committed, `go build` requires a prior frontend build.
- If `web/static` is committed, generated assets can become stale.
- Browser-history React routes require Go static fallback support and should not
  be introduced silently.
- Adding broader UI features during the port can obscure regressions in the
  current auth and group flows.

Keep the first migration narrow: replace the implementation technology, preserve
the current behavior, and leave product expansion for follow-up work.
